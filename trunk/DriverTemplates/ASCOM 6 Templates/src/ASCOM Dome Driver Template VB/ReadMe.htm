ASCOM Dome Driver (VB)



You have just created the skeleton of an ASCOM Dome driver in VB. It produces an in-process (assembly) based driver.

Prior to developing your first driver, please familiarize yourself with the developer information we've provided at the ASCOM Initiative web site (internet required).

You must do the following in order to complete your implementation:

  1. Run Visual Studio as Administrator. This is needed to allow it to register the driver.

  2. Set the Debug configuration and build the template now. It should build without errors.

  3. Add a test project to the Solution. A test console application is provided. To use this:

  • Select the Solution and use the Context Menu to Add a new Project. Select “Test Application (CS)”.

  • Check that the Test Application is set to run as startup.

  • Edit Program.cs to uncomment the line:
    Dim id As String = ASCOM.DriverAccess.Dome.Choose("")

  1. Debug (F5) your solution for a quick check of your driver. It runs the chooser, select your driver – it should be in the chooser list and select Properties to run the setup dialog, even if you haven't yet implemented any of the properties and methods.

  2. Open the AssemblyInfo.cs and Driver.cs files, then show the Task List and select Comments. This will show you the places where you need to add or change things for your driver. Tasks include implementing the required properties and methods. See the ASCOM Dome spec.

  3. Customize the Setup Dialog (SetupDialogForm) to provide the settings and other controls for your driver.

  4. Add or uncomment code in your test program to exercise more of your driver, for example:

Dim dome As ASCOM.DriverAccess.Dome = New ASCOM.DriverAccess.Dome(id)

Console.WriteLine("name " + dome.Name)

Console.WriteLine("description " + dome.Description)

Console.WriteLine("DriverInfo " + dome.DriverInfo)

Console.WriteLine("driverVersion " + dome.DriverVersion)



  1. Exercise and debug your driver, adding functionality and test code to match.

Notes:

  • Successfully building the driver, as well as using regasm 32 and 64 on the assembly, registers it for both COM and ASCOM (the Chooser). See the code in the ASCOM Registration region of Driver.cs.

  • Doing a Clean for the project, as well doing a regasm -u on the assembly, unregisters it for both COM and ASCOM (the Chooser).

  • Place a breakpoint in your driver class constructor, then start debugging (go, F5). Your breakpoint will be hit as soon as the test script creates an instance of your driver (after selecting it in the Chooser). You can now single step, examine variables, etc. Please review the test application and feel free to make changes and additions to activate various parts of your driver during debugging. You can bypass the appearance of the Chooser by changing
    Dim id As String = ASCOM.DriverAccess.Dome.Choose("")
    Dim dome As ASCOM.DriverAccess.Dome = New ASCOM.DriverAccess.Dome(id)
    to
    Dim id As String = ASCOM.DriverAccess.Dome.Choose("")
    Dim dome As ASCOM.DriverAccess.Dome = New ASCOM.DriverAccess.Dome(“ASCOM.$safeprojectname$.Dome”)
    This is a great way to incrementally debug your driver's properties and methods.

  • The project uses the new Settings, as described in the ASCOM Settings Provider document. The ComPort setting is included as an example of setting the serial port name. Modify or remove this as required.

ASCOM Initiative



The ASCOM Initiative consists of a group of astronomy software developers and instrument vendors whose goals are to promote the driver/client model and scripting automation.

See the ASCOM web site for more information. Please participate in the ASCOM-Talk Yahoo Group.
